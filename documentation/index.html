Neural Nomads: The Language Agnostic Bot

Comprehensive Technical Documentation & Maintenance Guide

Bridging communication gaps in educational institutions with a state-of-the-art, multilingual AI assistant. This document provides a complete guide for developers to understand, set up, and maintain the project.

1.1 Problem Statement

University and college administrative offices are consistently strained by a high volume of repetitive student inquiries. Questions about deadlines, fees, forms, and procedures lead to long queues and increased workload for administrative staff. This issue is significantly compounded by language barriers, creating communication gaps that prevent students—especially those from diverse linguistic backgrounds—from getting quick, clear, and accurate answers to their critical questions.

1.2 Our Solution

The Language Agnostic Bot is a sophisticated, AI-powered chatbot designed to resolve this challenge. It provides an instant, 24/7, and reliable source of information for students. By ingesting all official college documents—such as PDFs, circulars, and notices—it creates a comprehensive knowledge base. The bot can understand and respond to queries in all 22 constitutional Indian languages, effectively eliminating communication barriers. Furthermore, it features an intuitive admin dashboard that empowers non-technical staff to easily manage the bot's knowledge, view analytics, and handle escalated queries, ensuring the system is both powerful and easy to maintain.

1.3 Key Features (USPs)

Full Multilingual Support

Text and voice support for all 22 constitutional languages using advanced translation and speech-to-text models.

Omnichannel Presence

A single knowledge base provides consistent answers across the college website, WhatsApp, and Telegram.

Plug & Play Integration

Effortless deployment on any website by simply embedding a single HTML script tag.

Advanced RAG Pipeline

Context-aware conversations with an intelligent fallback to ticket creation when an answer isn't available.

Self-Improving Knowledge Base

Automatically clusters user queries to identify trends and suggest new FAQs, continuously improving its accuracy.

Data Privacy Options

Flexible deployment with support for cloud APIs (Gemini/GPT) or private, on-premise models like LLaMA.

2.2 Core Components Explained

Flask Backend (`app.py`)

This is the core of the application. It's a Python Flask server that exposes all REST API endpoints for the frontend, handles chat requests via Socket.IO for real-time communication, and manages database interactions with SQLAlchemy.

Data Ingestion (`ingest.py`, `run_ingestion.py`)

A dedicated background service that continuously monitors the `data/` directory. When new documents are added, it uses `utils.py` to extract text (including OCR for scanned PDFs), splits the text into manageable chunks, generates vector embeddings using Google's models, and stores them in the ChromaDB vector database for efficient retrieval.

RAG Core (`rag_chat.py`)

This module contains the primary chat logic. When a user query is received, it queries the ChromaDB vector store to find the most relevant document chunks. It then constructs a comprehensive prompt, including the retrieved context and recent conversation history, and sends it to the Gemini LLM. It also manages the stateful fallback mechanism, offering to create a support ticket if it cannot find a confident answer.

Multilingual Engine (`translator.py`)

This crucial component makes the bot language-agnostic. It uses the powerful IndicTrans2 model from AI4Bharat to translate any incoming user query into English, which is the base language for the RAG pipeline. After the English response is generated by the LLM, this module translates it back into the user's original language, ensuring a seamless multilingual experience.

Database (`models.py`)

Defines the application's entire data schema using SQLAlchemy ORM. It includes tables for storing conversation logs, user-created FAQs, raised support tickets, and the overall bot configuration settings from the admin dashboard.

Automated FAQ Generation (`cluster_faq.py`)

An offline script that enhances the bot's knowledge. It ingests historical conversation logs, uses SentenceTransformers to create embeddings for user queries, and applies KMeans clustering to group similar questions. Each cluster is then sent to Gemini to synthesize a representative question-answer pair, which can be added to the FAQ database, allowing the bot to improve over time.

3.1 Prerequisites

Before you begin, ensure you have the following software installed on your system:

  • Python 3.9+ and Pip
  • A virtual environment tool (like `venv`)
  • Git for cloning the repository
  • Tesseract OCR Engine: This is a system-level dependency.
    • Official Installation Guide
    • On Debian/Ubuntu: `sudo apt-get install tesseract-ocr`
    • On macOS (via Homebrew): `brew install tesseract`
    • Ensure the Tesseract executable is in your system's PATH.

3.2 Initial Setup

Clone the repository and set up a Python virtual environment.

# 1. Clone the repository
git clone https://github.com/your-username/language-agnostic-bot.git

# 2. Navigate into the project directory
cd language-agnostic-bot

# 3. Create a Python virtual environment
python3 -m venv venv

# 4. Activate the virtual environment
# On macOS/Linux:
source venv/bin/activate
# On Windows:
# venv\Scripts\activate

3.5 Database Initialization

The application uses SQLite by default. The database file (`app.db`) will be created automatically in an `instance` folder the first time you run the Flask application. No manual database setup is required for local development.

3.6 Running the Application

The application consists of two main processes that need to be run in separate terminals: the data ingestion service and the main web server.

Step 1: Start the Data Ingestion Service

This service watches the `data/` folder for new documents to process. Before running it, create a `data` directory in the project root and place some sample PDF files inside it.

# In your first terminal (make sure venv is active)
# Create the data directory if it doesn't exist
mkdir -p data

# Run the ingestion script
python run_ingestion.py

This script will run indefinitely, checking for new files every 60 seconds.

Step 2: Start the Main Flask Application

This command starts the web server, making the chatbot and admin dashboard accessible.

# In your second terminal (make sure venv is active)
flask run --port 5001

Your application is now running and accessible at http://127.0.0.1:5001.

4.1 System Monitoring

Checking Service Status: The simplest way to monitor the system is to watch the console output of the two running processes. The `run_ingestion.py` terminal will print messages when it checks for files or processes them. The `flask run` terminal will show API requests, database queries, and any runtime errors.

Database Health: The application uses a SQLite database file located at `instance/app.db`. You can use a free tool like DB Browser for SQLite to open this file and visually inspect the tables (`conversations`, `tickets`, `faqs`) to ensure data is being saved correctly.

Vector Store Health: The ChromaDB vector store is located in the `db/` directory. If this directory contains files like `chroma.sqlite3` and several `.bin` files, it means the ingestion service has successfully created and populated the vector collection.

4.2 Common Errors & Debugging Steps

Error 1: `ModuleNotFoundError: No module named '...'`

Cause: A required Python package is missing from your virtual environment, or the environment is not activated.

Solution:

  1. Make sure your virtual environment is active (your command prompt should be prefixed with `(venv)`).
  2. Run `pip install -r requirements.txt` again to ensure all dependencies are installed.

Error 2: `google.auth.exceptions.DefaultCredentialsError`

Cause: The application cannot find your Google API key. This usually means the `.env` file is missing, misnamed, or the `GOOGLE_API_KEY` variable is not set correctly.

Solution:

  1. Ensure a file named exactly `.env` exists in the project's root directory.
  2. Verify that the file contains the line `GOOGLE_API_KEY="your-key-here"` with your actual key.
  3. Restart the Flask server and ingestion script after creating or modifying the `.env` file.

Error 3: `pytesseract.TesseractNotFoundError`

Cause: The Tesseract OCR engine is not installed on the operating system, or its location is not included in the system's PATH environment variable.

Solution: Follow the installation instructions for your OS (see Prerequisites section). After installation, you may need to restart your terminal or your computer for the PATH changes to take effect.

4.3 Backup and Recovery

Regular backups are crucial. The two most important components to back up are the application database and the vector store.

Database Backup

# Create a backups directory
mkdir -p backups
# Copy the database file with a timestamp
cp instance/app.db backups/app_backup_$(date +%Y%m%d%H%M%S).db

Vector Store Backup

The vector store contains the processed embeddings of all your documents. Re-creating it can be time-consuming, so it's wise to back up the entire `db/` directory.

# Archive the entire db directory
tar -czvf backups/vector_store_backup_$(date +%Y%m%d%H%M%S).tar.gz db/

6. Code Modules Explained

app.py

The main entry point of the web application. It initializes the Flask app, configures CORS, Socket.IO, and the database. It defines all API endpoints for managing documents, FAQs, tickets, and handling chat requests. It also serves the frontend admin panel and the embeddable chat widget.

rag_chat.py

The core logic for the Retrieval-Augmented Generation (RAG) process. This module handles user queries by retrieving relevant context from the ChromaDB vector store, constructing a prompt with conversation history, invoking the Gemini LLM for a response, and managing the fallback logic to raise a support ticket when information is unavailable.

ingest.py

Responsible for processing source documents. It reads files from the `data/` directory, uses functions from `utils.py` to extract text, splits the text into chunks, generates embeddings using the Google Generative AI embeddings model, and adds these embeddings to the ChromaDB collection for future retrieval.

translator.py

Provides the multilingual capabilities. It contains functions to translate text from an Indian language to English (using the `xx2en` model) before it enters the RAG pipeline, and to translate the English response back to the user's original language (using the `en2xx` model), powered by AI4Bharat's IndicTrans2.

models.py

Defines the database schema using Flask-SQLAlchemy. It contains the data models for `Conversation` logs, `FAQ` entries, bot `BotConfig` settings, and support `Ticket` information, mapping Python classes to database tables.

utils.py

A collection of utility functions, primarily for text extraction. It includes functions to extract text from standard PDFs using `PyPDF2` and a fallback function to perform Optical Character Recognition (OCR) on scanned or image-based PDFs using `pytesseract` and `pdf2image`.

cluster_faq.py

An offline script for continuous improvement of the knowledge base. It reads historical user queries, clusters them based on semantic similarity using `sentence-transformers` and `scikit-learn`, and then uses the Gemini LLM to synthesize a canonical question and answer for each cluster, which can then be added as a new FAQ.
